Engineering Standards : Asset Tagging Requirements

Created by Ryan Fisch, last modified on Apr 14, 2017

Resource Tagging Policy

This document sets for the expectation for tag management within the Placester AWS environment. All taggable assets should have the minimum of these tags in place. Tags are a key-value pair that allows an individual to identify an attribute of AWS with a specific value. The value of tags are to provide better reporting for utilization and cost management.

*Note: tags are case sensitive on Key and Values. Also they are limited to 128 and 256 respectively.

Assets with Tag Support:

  • AMI

  • Customer Gateway

  • DHCP Option

  • EBS Volume

  • EC2 Instance

  • Network ACL

  • Network Interface

  • Reserved Instance

  • Route Table

  • Spot Instance Request

  • Security Group

  • Subnet

  • Virtual Private Gateway

  • Virtual Private Cloud

  • VPN Connection

General Restrictions

  • Maximum number of tags per resource – 10  

  • Maximum key length – 127 Unicode characters  

  • Maximum value length – 255 Unicode characters  

  • Tag keys and values are case sensitive  

  • Tag keys must be unique per resource  

  • Do not use the “aws:” prefix in your tag name or values because it is reserved for AWS use


The following are a series of tags, their definition, and expected / example values that should be mapped for each asset in AWS.

Required Tags

Tag Name: TeamOwner

Description: Defines the primary team that “owns” this asset

Values:

  • Infrastructure (EngOps)

  • Web (SAS)

  • Data (Archimedes)

  • Search (Archimedes)

  • Services (Monoceros)

  • Engineering (email only)


These tags are used by PagerDuty trigger scripts to route Monit alerts to the correct Team/Escalation path (in parenthesis above) as defined by the following case statement in internal.aws.plcstrint.com:/placester/pagerduty/get_service_key.sh

case $TeamOwner in

       Infrastructure )PAGERDUTY_SERVICE_KEY=$INFRASTRUCTURE_KEY;;

       Web )           PAGERDUTY_SERVICE_KEY=$WEB_KEY;;

       Data )          PAGERDUTY_SERVICE_KEY=$DATA_KEY;;

       Search )  PAGERDUTY_SERVICE_KEY=$DATASERVICES_KEY;;

       Services )      PAGERDUTY_SERVICE_KEY=$SERVICES_KEY;;

       Engineering)    PAGERDUTY_SERVICE_KEY="";;

       * )             PAGERDUTY_SERVICE_KEY=$HOTPOTATO_KEY

esac


Tag Name: Owner


Description: This is to be used in the pl-internal account (colloquially referred to as "the sandbox"). Defines the primary individual that “owns” this asset.

Any asset in the pl-internal account that does not have an owner can and will be shut down and terminated without notice to control costs.  Assets that are consuming financial resources that have an Owner tag can still be shut down outside of business hours without notice but will not be destroyed.   Owner tag should be used in association with TeamOwner = Engineering and the value should be your personal AWS userid.

  Optional Tags

Tag Name: Domain


The domain tag should represent the service or application generalized context or domain area. Examples: etl, sites, billing, subscription, accounts

Tag Name: Service

The service tag should represent a more specific sub system, microservice, application specific detail.  Example could be a micro-service name, lambda function name, application module, et cetera.
