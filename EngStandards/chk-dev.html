Engineering Standards : Service Design / Review Checklist

Created by Jerry Shaughnessy, last modified on Oct 27, 2017
Architecture & Design Considerations
  • All entry points and trust boundaries are identified
  • Input validation for all external callers
  • Design assumes malicious callers
  • Design address potential cross-site scripting issues
  • Design does not rely on client-side validation
  • Clear division of external and internal interfaces
  • API is following REST (or GraphQL rules)
  • All remote resources are called asynchronously whenever possible
  • System is designed to gracefully degrade.
  • Version strategy defined and implemented
  • Service is stateless
Documentation
  • Cost analysis completed and included with design
  • Implementation matches the design documentation
  • Contracts for API are defined and published
  • A core READ.MD file exists in the code and is sufficient to get a new developer up and running with a basic understanding of the system/service functionality and interactions
  • There are examples of all resources
  • Initial design documents exist with sequence diagrams
Monitoring & Logging
  • All exceptions are logged
  • At a minimum, all boundary methods have measures applied to them
  • Error codes are used throughout
  • Logging can be configured and level changed without requiring a redeployment
  • No sensitive data is logged
  • Correlation ID's are used whenever possible
Communication & External Resources
  • SSL everywhere - all the time
  • Return fields can be limited / sorted / filtering
  • Any necessary throttling has been identified and implemented
  • JSON only responses
  • Latency and response SLA's identified and documented
  • Bulkhead strategy employed against external resources
  • Circuit-break strategy employed against external resources
  • GET, HEAD, PUT, DELETE, OPTIONS and TRACE are all idempotent
Naming & Style
  • Consistent and Intuitive Naming used
  • Are there any undocumented assumptions
  • Are the proper HTTP response codes being returned
Performance & Testing
  • Consistent and Intuitive Naming used
  • No unnecessary dependencies
  • Unit tests exist and code coverage is >90%
  • Can this code gracefully handle external errors (e.g. - network, db, etc.).
  • Are all resources released when they are no longer needed
  • Has the code /service performance been validated over the desired target range
  • Has any auto-scaling functionality been tested and documented
